Registrar Checkout Push Module
==============================

Purpose
-------
This document describes the JSON payload that `checkoutApi` sends to each registrar's
`api_endpoint` when a customer places an order through the checkout flow.

Transport
---------
- Method: `POST`
- Content-Type: `application/json`
- Success rule: any HTTP `2xx` response is treated as accepted
- Failure rule: any non-`2xx` response is treated as rejected for that attempt and may be retried

Canonical fields to rely on
---------------------------
The receiving registrar service or AI worker should treat these top-level fields as the primary
contract:

- `order_reference`: the public order reference generated by checkoutApi
- `first_name`: customer first name
- `last_name`: customer last name
- `phone`: customer phone number
- `email`: customer email
- `company_name`: customer company or organisation name when provided
- `kra_pin`: customer KRA PIN when provided
- `street_address`: customer street address
- `city`: customer city
- `state`: customer state or region
- `postcode`: customer postal or ZIP code
- `country`: customer country
- `domain_name`: requested domain in normalized lowercase form
- `domain_extension`: domain extension when available
- `plus`: high-level add-on family label such as `Hosting`, `Emails`, `Servers`, or `Security`
- `type`: service type label when applicable, for example `Email Hosting`
- `package`: chosen package label when applicable
- `registrar_name`: selected registrar display name
- `price_ksh`: quoted order price in KES when known
- `currency_code`: order currency when known
- `period`: human-readable billing period such as `Monthly`, `Yearly`, or `12 months`

This field order is intentional and matches the app-side checkout builders and the checkout API
normalization layer so registrar implementations can mirror one stable customer payload shape.

Payload example
---------------
{
  "order_reference": "YT6E45DN5N",
  "domain_name": "glasst.co.ke",
  "domain_extension": ".co.ke",
  "first_name": "Jane",
  "last_name": "Doe",
  "phone": "+254700000000",
  "email": "hello@example.com",
  "company_name": "Example Ventures Limited",
  "kra_pin": "A123456789Z",
  "street_address": "Westlands Road 14",
  "city": "Nairobi",
  "state": "Nairobi County",
  "postcode": "00100",
  "country": "Kenya",
  "plus": "Emails",
  "type": "Email Hosting",
  "package": "Mail Team",
  "registrar_name": "Truehost Cloud Limited",
  "price_ksh": 450,
  "currency_code": "KES",
  "period": "Monthly"
}

Important notes
---------------
- `order_reference` should be stored by the registrar side and used as the external reference for
  the order.
- The structured customer profile fields should be preserved exactly as received.
- The order summary should be read from `domain_name`, `domain_extension`, `plus`, `type`,
  `package`, `registrar_name`, `price_ksh`, `currency_code`, and `period` directly. No extra
  combined fields such as `full_name`, `customer_location`, `target_service`, or internal
  selection fields are required.

Receiver design guidance
------------------------
1. Use `order_reference` as the idempotency key and thread key for your internal order workflow.
2. Validate `domain_name`, the structured customer profile fields, and the simple purchase summary
   fields before
   accepting the order.
3. Route by `plus`, `type`, `package`, and `period` to select the exact fulfillment path.
4. Keep a copy of the raw payload so human support and AI agents can inspect the original request.
5. Return non-`2xx` for validation failures or rejected orders so checkoutApi can record the failure
   correctly.

Recommended success response
----------------------------
If the registrar accepts the order, return HTTP `200`, `201`, or another `2xx` response.

Recommended JSON body:
{
  "accepted": true,
  "reference_id": "REG-2026-000123",
  "message": "Order accepted and queued for fulfillment."
}

Response handling notes
-----------------------
- `reference_id` or `referenceId` will be stored by checkoutApi as `registrar_reference_id`
- Any other JSON fields may still be returned for your own diagnostics
- A plain-text body is also acceptable, but JSON is preferred

Recommended failure response
----------------------------
If the registrar cannot accept the order, return a non-`2xx` response such as `400`, `409`, or
`422`.

Recommended JSON body:
{
  "accepted": false,
  "message": "Package code MAIL_TEAM is not active for this registrar."
}
