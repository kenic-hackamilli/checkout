Registrar Checkout Push Module
==============================

Purpose
-------
This document describes the JSON payload that `checkoutApi` sends to each registrar's
`api_endpoint` when a customer places an order through the checkout flow.

Transport
---------
- Method: `POST`
- Content-Type: `application/json`
- Success rule: any HTTP `2xx` response is treated as accepted
- Failure rule: any non-`2xx` response is treated as rejected for that attempt and may be retried

Canonical fields to rely on
---------------------------
The receiving registrar service or AI worker should treat these top-level fields as the primary
contract:

- `order_reference`: the public order reference generated by checkoutApi
- `full_name`: customer name
- `email`: customer email
- `phone`: customer phone number
- `domain_name`: requested domain in normalized lowercase form
- `domain_extension`: domain extension when available
- `target_service`: machine-readable service code such as `domain_registration_only`,
  `email_hosting`, `shared_hosting`, `vps_hosting`, `wordpress_hosting`, or `ssl`
- `package_name`: chosen package label when applicable
- `package_code`: package code when available
- `billing_cycle`: normalized cycle such as `monthly`, `yearly`, `custom`, or `flexible`
- `billing_period_months`: billing length in months when known
- `quoted_price_ksh`: quoted order price in KES when known

Important notes
---------------
- `order_reference` should be stored by the registrar side and used as the external reference for
  the order.
- `target_service` is the canonical routing field. Do not infer the service from `package_name`
  when `target_service` is present.
- `package_name` is the canonical human-readable label for the chosen package.
- `selection_snapshot_json` is supporting context. It may contain extra descriptive fields for UI
  or AI reasoning, but the registrar workflow should prefer top-level fields when both exist.
- `selected_offering_label` is intentionally not sent because `package_name` already covers that
  meaning.

Receiver design guidance
------------------------
1. Use `order_reference` as the idempotency key and thread key for your internal order workflow.
2. Validate `domain_name`, `target_service`, and customer contacts before accepting the order.
3. Route by `target_service`, then use `package_name`, `package_code`, and pricing fields to select
   the exact fulfillment path.
4. Keep a copy of the raw payload so human support and AI agents can inspect the original request.
5. Return non-`2xx` for validation failures or rejected orders so checkoutApi can record the failure
   correctly.

Recommended success response
----------------------------
If the registrar accepts the order, return HTTP `200`, `201`, or another `2xx` response.

Recommended JSON body:
{
  "accepted": true,
  "reference_id": "REG-2026-000123",
  "message": "Order accepted and queued for fulfillment."
}

Response handling notes
-----------------------
- `reference_id` or `referenceId` will be stored by checkoutApi as `registrar_reference_id`
- Any other JSON fields may still be returned for your own diagnostics
- A plain-text body is also acceptable, but JSON is preferred

Recommended failure response
----------------------------
If the registrar cannot accept the order, return a non-`2xx` response such as `400`, `409`, or
`422`.

Recommended JSON body:
{
  "accepted": false,
  "message": "Package code MAIL_TEAM is not active for this registrar."
}
